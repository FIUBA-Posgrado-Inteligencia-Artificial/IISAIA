Tres conceptos, mínimo viable

1commit

Una foto del repo en un momento. Guarda QUÉ cambió (el diff), QUIÉN lo hizo, CUÁNDO, y POR QUÉ (mensaje del commit).

Regla práctica: commits chicos y frecuentes. Si no se describe en una sola oración corta, probablemente sean dos.

2branch

Una línea paralela de commits. main es la principal. Una branch propia (feature/<descripcion>) no contamina main mientras experimentás.

Mientras la branch está abierta podés romper, deshacer, reintentar — nadie más se entera.

3pull request

La propuesta de mergear tu branch a main. No es un comando — es un evento en GitHub. Tiene título, descripción, y queda visible para el equipo.

Hasta que no se mergea, la branch existe pero main no tiene esos cambios.

Tres reveals — uno por concepto.

Refresher de los conceptos base. Suposición: el alumno los vio antes en S04 pero conviene retomarlos antes de las skills que operan sobre git.

"Tres conceptos mínimos. El primero: commit. Una foto del repo en un momento. Guarda qué cambió, quién, cuándo, y por qué — el mensaje del commit. La práctica es hacer commits chicos y frecuentes: si no se puede describir en una sola oración corta, probablemente sean dos commits."

"Segundo: branch. Una línea paralela de commits. Main es la línea principal del proyecto. Cuando arrancan algo nuevo, abren una branch propia con el formato feature barra descripcion. Mientras está abierta, pueden romper, deshacer, reintentar — nadie más se entera."

"Tercero: pull request. La propuesta de mergear tu branch a main. No es un comando: es un evento en GitHub. Lo abren desde la UI, le ponen título y descripción, y queda visible. Hasta que no se mergea, la branch existe pero main no tiene esos cambios."

El flujo concreto con Superpowers

Antes de arrancar la sesión 
git checkout main
git pull
git checkout -b feature/<descripcion>
Durante el trabajo

Superpowers, vía subagent-driven-development, commitea por tarea automáticamente. Vos revisás los commits entre tareas. Si querés modificar algo a mano, el flujo no te bloquea.

Cuando el feature está listo 
git push origin feature/<descripcion>

Después, en la UI de GitHub: abrís el PR. Título y descripción claros. Superpowers ya generó los commits con buenos mensajes; vos contás la historia general en la descripción del PR.

Tres reveals — uno por fase.

El workflow concreto. Tres fases: setup, trabajo, cierre.

"Antes de arrancar la sesión: checkout main, pull para traer lo último, y checkout barra b feature barra descripcion. Empezás desde main actualizado y abrís tu branch. Esos tres comandos son el setup de cada sesión."

"Durante el trabajo: Superpowers, vía subagent-driven-development, commitea por tarea automáticamente. Eso lo van a ver en detalle en la próxima sección. Tu trabajo es revisar los commits que va generando entre tareas y aceptarlos. Si querés modificar algo a mano, podés — el flujo no te bloquea."

"Cuando el feature está listo, después de que verification-before-completion pasó: push de la branch al origin. Y después, en la UI de GitHub, abrís el PR. Título y descripción claros. Una nota: Superpowers ya generó los commits con buenos mensajes durante la sesión; lo que ustedes hacen al cerrar es contar la historia general en la descripción del PR."

Por qué este patrón encaja

Aislamiento por sesión Cada sesión tiene su branch. El contexto del próximo trabajo no se contamina con cambios a medio terminar.
PR como contrato visible Revisable por humanos antes de que el código entre a main. Queda registro de qué se discutió y por qué se aprobó.
main siempre estable Si algo de la sesión sale mal, descartás la branch y volvés a empezar sin romper nada del proyecto.
Sin operaciones complicadas No necesitás stash, rebase complicado, ni cherry-pick. Sólo branch, commit, push, PR.

Honest disclosure — using-git-worktrees

Existe una skill que automatiza un workflow más avanzado: trabajar en varias branches en paralelo, cada una en su propio folder físico del mismo repo. No la usamos acá — agrega fricción cognitiva sobre git base, y branch + PR alcanza para el trabajo final y para la mayoría del trabajo de equipo real.

Dos reveals — razones del encaje + worktrees disclosure.

Cierre de §8. Por qué branch + PR es buen match para Superpowers + qué skill alternativa existe pero no la usamos.

"Cuatro razones por las que este patrón se lleva bien con Superpowers. Aislamiento por sesión — cada sesión tiene su branch, el contexto del próximo trabajo no se contamina con cambios a medio terminar. El PR queda como contrato visible — revisable por humanos antes de que el código entre a main. Main siempre permanece estable — si algo de la sesión sale mal, descartás la branch y volvés a empezar sin romper nada del proyecto. Y no necesitás stash, rebase complicado, ni cherry-pick: solo branch, commit, push, PR."

"Antes de cerrar la sección, una mención honesta. Existe una skill llamada using-git-worktrees que automatiza un workflow más avanzado. Worktrees permite trabajar en varias branches en paralelo, cada una en su propio folder físico del mismo repo. No la usamos en este curso porque agrega fricción cognitiva sobre git base, y el flujo simple de branch más PR alcanza para el TP y para la mayoría del trabajo de equipo real. Si en algún momento la quieren explorar, está en el cache de Superpowers."
