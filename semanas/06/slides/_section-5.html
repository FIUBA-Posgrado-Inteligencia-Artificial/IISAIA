¿Cómo es que el LLM usa tools que se publicaron después de su cutoff?

Hook de §5. La pregunta que el alumno arrastra desde S04 — el modelo se congela en el entrenamiento, ¿cómo usa algo que no vio? Sin reveals, la pregunta queda flotando antes de la respuesta.

"Pongámoslo así. Conectás un MCP Server que alguien escribió la semana pasada — no existía cuando este modelo se entrenó. Y el modelo lo usa sin problema, de una. Pero el modelo está congelado, su conocimiento tiene una fecha de corte, un cutoff."

"Entonces, ¿cómo es que el LLM usa tools que se publicaron después de su cutoff? Nunca los vio, no los memorizó, no sabe que existen. Esta pregunta es la pieza más elegante de MCP, y todo este bloque es la respuesta. Vamos a desarmarla."

Dos mecanismos, una conexión.

La respuesta no es magia: son dos pasos del protocolo, bien separados.

cuando arranca · una vez

Capability negotiation

Durante el initialize, cliente y server se cuentan qué versión del protocolo hablan y qué saben hacer.

en la conversación · cada vez

Discovery

Con métodos */list, el cliente le pregunta al server qué tools, resources y prompts tiene ahora.

Y si el catálogo cambia mientras hablás, hay un tercer mecanismo: notifications. Lo vemos al final.

Tres reveals — capability negotiation, discovery, mención de notifications.

Separa los dos mecanismos antes de profundizar cada uno. La distinción clave es el "cuándo": negotiation pasa una vez al arrancar, discovery pasa durante la conversación.

"La respuesta vive en dos mecanismos que conviene separar bien. El primero: capability negotiation. Pasa una sola vez, cuando el host arranca el server, durante un mensaje que se llama initialize. Ahí cliente y server se presentan: yo hablo esta versión del protocolo, soporto estas capabilities; el otro responde igual."

"El segundo: discovery. Pasa durante la conversación, cada vez que hace falta. El cliente le pregunta al server qué tiene disponible con métodos que terminan en list — tools slash list, resources slash list, prompts slash list. La palabra clave es ahora: le pregunta qué tiene en este momento, no qué tenía cuando se programó."

"Y hay un tercero, más chico, para cuando el catálogo cambia en vivo: las notifications. Lo dejamos para el cierre del bloque. Vamos al primero."

El handshake.

Como dos extraños que se presentan: qué versión hablás, qué sabés hacer.

Cliente dice

{
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "sampling": {},
      "elicitation": {}
    },
    "clientInfo": { "name": "claude-code" }
  }
}

Server responde

{
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": { "subscribe": true },
      "prompts": {}
    },
    "serverInfo": { "name": "playwright-mcp" }
  }
}

MCP es un protocolo stateful: esa negociación queda registrada para toda la sesión. Cada uno sabe exactamente qué puede pedirle al otro.

Tres reveals — mensaje del cliente, respuesta del server, la línea de stateful.

El handshake concreto. No hace falta que lean cada campo: la idea es que se presentan mutuamente. La analogía de los dos extraños evita que JSON-RPC suene a magia.

"El handshake, concreto. El cliente abre con un mensaje initialize. Le dice al server tres cosas: qué versión del protocolo habla — ese protocolVersion con la fecha —, qué capabilities soporta él como cliente — acá sampling y elicitation, las primitives del cliente que vimos en el bloque anterior — y quién es."

"El server responde con la misma forma: yo hablo esa misma versión, y estas son mis capabilities. Fíjense en una en particular: tools, con listChanged en true. Eso significa 'tengo tools, y además te voy a avisar si cambian'. Esa banderita es la que habilita las notifications del final."

"Y la propiedad clave: MCP es stateful. Esa negociación no se tira después del primer mensaje — queda registrada para toda la sesión. Cliente y server saben en todo momento qué se pueden pedir mutuamente. No re-negocian en cada vuelta."

El catálogo.

Después del handshake, el cliente pregunta qué hay con */list. La respuesta no es una lista de nombres.

Cliente pide

{
  "method": "tools/list"
}

Server devuelve un catálogo con metadata

{
  "result": {
    "tools": [
      {
        "name": "browser_navigate",
        "description": "Navigate the browser to a URL",
        "inputSchema": {
          "type": "object",
          "properties": { "url": { "type": "string" } },
          "required": ["url"]
        }
      }
    ]
  }
}

Esa metadata — name, description, inputSchema — se inyecta al system context del modelo. Antes de razonar sobre tu pregunta, el modelo ya tiene en la mesa la descripción exacta de cada tool.

Tres reveals — request, response con metadata, la línea del system context.

El catálogo con metadata. El punto pedagógico clave: la respuesta no son nombres pelados — trae description e inputSchema, y eso es lo que el modelo lee para decidir cuándo invocar. Esa metadata viaja al system context.

"Después del handshake, el cliente descubre qué hay. Manda un tools slash list — un mensaje cortito, sin parámetros. 'Decime qué tools tenés.'"

"Y el server le devuelve un catálogo. Pero ojo, no es una lista de nombres pelada. Cada tool viene con metadata: el nombre, una description en prosa de qué hace, y un inputSchema que dice qué parámetros toma y de qué tipo. Acá ven browser_navigate, de Playwright, con su descripción y su único parámetro url, requerido, de tipo string."

"Y acá está el mecanismo completo: esa metadata se le inyecta al modelo como parte del system context de la conversación. Antes de razonar sobre lo que vos le pediste, el modelo ya tiene en la mesa la descripción exacta de cada herramienta que dispone. La description que escribió el autor del server es literalmente lo que el modelo lee para decidir cuándo usarla. Por eso un server escrito hoy es usable hoy: su catálogo entra al contexto en tiempo de conversación, sin tocar el modelo."

El LLM no memoriza tools.
Los descubre en tiempo de conversación.

Por eso cualquier server nuevo, escrito hoy, es usable hoy — sin reentrenar el modelo, sin actualizarlo, sin que el vendor del LLM se entere.

El insight articulado en una frase. Slide-poster sin adornos. Esta es la frase que más vale la pena que se lleven de toda la sección.

"Si se llevan una sola frase de este bloque, que sea esta. El LLM no memoriza tools. Los descubre en tiempo de conversación."

"Y la consecuencia es enorme: cualquier server nuevo, escrito hoy, es usable hoy. Sin reentrenar el modelo, sin esperar una actualización, sin que la empresa que hace el LLM tenga que enterarse siquiera de que ese server existe. La integración funciona porque hay un protocolo común que dice 'así te describís, y así te invocan'. El descubrimiento reemplaza a la memorización."

Y si cambia en vivo.

El catálogo no es una foto inicial. Acá entra el tercer mecanismo.

El server avisa

{
  "method":
    "notifications/tools/list_changed"
}
→

El cliente reacciona

Vuelve a pedir tools/list y actualiza el contexto del modelo en tiempo real.

Cubre servers que exponen tools según los roots que reciben, o que agregan tools cuando el usuario amplía permisos. La discovery es dinámica, no una foto.

Tres reveals — la notificación del server, la reacción del cliente, el caption de discovery dinámica.

Notifications cierra el mecanismo. La banderita listChanged del handshake era la habilitación; acá se usa. El punto: el catálogo puede cambiar a mitad de sesión y el contexto se actualiza solo.

"¿Se acuerdan del listChanged en true del handshake? Acá se cobra. Cuando el server declaró esa capability, se reserva el derecho de avisar en cualquier momento que su catálogo cambió. Manda una notificación — notifications slash tools slash list_changed. Sin respuesta esperada, es un aviso."

"Y el cliente reacciona: vuelve a pedir tools slash list, trae el catálogo nuevo, y actualiza el contexto del modelo en tiempo real. El modelo, en la próxima vuelta, ve el set actualizado."

"¿Cuándo pasa esto en la práctica? Un server que expone tools distintos según los directorios que le pasaron como scope. Un server con auth por scopes que suma tools cuando el usuario amplía permisos a mitad de sesión. Un server cuyo catálogo depende del estado de un proceso externo. La idea de fondo: discovery es dinámica, no una foto que se saca al arrancar."
