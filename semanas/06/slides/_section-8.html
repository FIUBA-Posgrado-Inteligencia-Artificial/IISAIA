Un remoto se agrega distinto.

Mismo comando base, pero apunta a una URL — y la auth es OAuth.

local · Stdio (Playwright)

$ claude mcp add playwright npx @playwright/mcp@latest
en el JSON"type": "stdio", command + args
arranqueun comando que corre local
authninguna — vive en tu máquina

remoto · Streamable HTTP (Linear)

$ claude mcp add --transport http linear-server https://mcp.linear.app/mcp
en el JSON"type": "http", url
arranqueuna URL — el server ya corre afuera
authOAuth en el browser
en Claude Code: /mcp → se abre el browser, login en Linear → el token queda guardado (se renueva solo)

El corazón de las dos secciones: el contraste local vs remoto. Local = un comando que arranca un proceso (Stdio). Remoto = una URL a un server que ya corre afuera (HTTP) + OAuth. Materializa los dos transports de §3.

"Acá está la diferencia, y es el sentido de hacer estos dos demos juntos. A la izquierda, el local de recién: claude mcp add, y un comando que arranca un proceso en tu máquina. En el JSON queda type stdio, con command y args. No hay auth porque el server vive con vos."

"A la derecha, el remoto: claude mcp add, pero con transport http, y en vez de un comando, una URL — la de Linear. El server ya está corriendo en internet; vos solo apuntás a él. En el JSON queda type http, con url. Y como es un servicio de terceros con tus datos, necesita autenticación."

"Esa auth es OAuth, y se resuelve adentro de Claude Code: corrés barra mcp, se abre el browser, te logueás en Linear y das permiso, y el token queda guardado de forma segura, renovándose solo. No pegás ningún token a mano. Esto es exactamente Stdio versus Streamable HTTP, los dos transports que vimos en la arquitectura, ahora en concreto."

Dos casos sobre un server remoto: lectura (buscar y resumir issues) y acción (crear un issue). El alumno tiene que ver que es el mismo patrón que Playwright, con datos que viven afuera.

"Al vivo, mismo molde que antes. Primero lectura: le pido que busque en Linear los issues abiertos que tengo asignados y me los resuma. El agente usa los tools de Linear para buscar, leer los issues y devolver el resumen — los datos vienen del workspace, no de la máquina."

"Después, acción: le pido que cree un issue nuevo. El agente llama el tool de crear y el issue aparece en Linear, en vivo. Mismo patrón que el form de Playwright — cambia el dominio, no el intercambio."

Dos servers, dos transports, un patrón.

Playwright · local · Stdio

Corre en tu máquina, como subproceso. Sin red, sin auth. Lo arranca un comando.

Linear · remoto · HTTP

Vive en internet, hosteado. Te conectás por red, con OAuth. Lo señalás con una URL.

Cambia dónde vive el server y cómo te conectás. El intercambio es idéntico: el server expone, el LLM descubre, invoca, el resultado vuelve.

Cierre del bloque de demos y del arco de los dos transports. La conclusión que se llevan: el transport (local/remoto) cambia la plomería de conexión, pero el patrón de intercambio MCP es el mismo. Esto cierra §3 (dos transports) con evidencia.

"Y este es el cierre de los dos demos. Pusimos lado a lado un server local y uno remoto. Playwright: local, Stdio, corre en tu máquina, sin auth, lo arranca un comando. Linear: remoto, HTTP, vive en internet, OAuth, lo señalás con una URL."

"Lo que cambia entre los dos es dónde vive el server y cómo te conectás — la plomería. Pero el intercambio, lo que pasa una vez conectados, es idéntico: el server expone sus capacidades, el modelo las descubre, las invoca, y el resultado vuelve a la conversación. Un solo patrón, para cualquier server, viva donde viva. Eso es MCP."
