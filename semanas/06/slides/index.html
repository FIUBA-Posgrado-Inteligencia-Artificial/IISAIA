Dos precedentes históricos del mismo movimiento.

1991

HTTP

HyperText Transfer Protocol

browsers × servers

Antes: cada browser hablaba con cada server por su cuenta. Después de HTTP, cualquier browser habla con cualquier server, sin coordinación entre autores. Internet se vuelve posible.

2016

LSP

Language Server Protocol

IDEs × lenguajes

Antes: cada IDE × cada lenguaje, autocompletado a mano. Después de LSP, cada lenguaje publica un server, cada IDE lo consume. La matriz colapsa.

MCP es el mismo movimiento, para agentes × capacidades. No inventa nada — aplica una idea conocida a un dominio nuevo.

Dos reveals — primero LSP, después la línea de cierre con MCP.

Dos precedentes históricos del mismo patrón, revelados de a uno. Ubican a MCP en una tradición — no es invento de Anthropic, es aplicar a agentes una idea que ya resolvió problemas análogos.

"Este movimiento que acabamos de ver — la matriz que se colapsa por un protocolo común — no es nuevo. Ya pasó dos veces en la historia reciente del software, y vale la pena ubicarlo donde lo vieron antes. Primero, HTTP. Año 91. HyperText Transfer Protocol. Antes de HTTP cada browser hablaba con cada servidor por su cuenta — coordinaban formatos, encodings, mecanismos. Después de HTTP, cualquier browser habla con cualquier server. Internet se vuelve posible. Esa es la propiedad del protocolo común: descomprime una matriz de coordinación humana en infraestructura compartida."

"Segundo, LSP — Language Server Protocol. 2016, Microsoft. Resolvió exactamente el mismo problema en el mundo de los IDEs. Antes, cada IDE — VS Code, Vim, Emacs, Sublime — implementaba autocompletado y go-to-definition para cada lenguaje — Python, Go, Rust, TypeScript — a mano. N por M. Insostenible. Después de LSP, cada lenguaje publica un language server, cada IDE consume cualquier server. N más M. Y eso es lo que hace que hoy cualquier IDE moderno tenga autocompletado decente en cualquier lenguaje, sin que el equipo del IDE escriba código específico para cada uno."

"MCP es el mismo movimiento, aplicado a agentes y capacidades. No inventa nada conceptualmente. Aplica una idea conocida — protocolo común — a un dominio donde el problema apareció hace poco."

Dos capas.

MCP se divide en data layer (qué se intercambia) y transport layer (por dónde).

capa interna

Data layer

Define qué se intercambia: estructura de mensajes, métodos, primitives, lifecycle, notifications. Wire format: JSON-RPC 2.0.

capa externa

Transport layer

Define por dónde viajan esos mensajes: canal físico y autenticación.

El mismo set de mensajes funciona idéntico sobre cualquier transporte. No tenés que aprender JSON-RPC para usar MCP — sólo saber que está abajo.

Separación canónica data layer vs transport layer. La razón pedagógica importante: el mensaje es el mismo independiente del canal. Eso es lo que hace que el protocolo viaje entre transports.

"MCP se divide internamente en dos capas, y vale separarlas mentalmente. La data layer es la capa interna: define qué se intercambia entre cliente y servidor. Mensajes, métodos, las primitives que vamos a ver en el próximo bloque, el lifecycle de la conexión, las notificaciones. Wire format JSON-RPC 2.0 — formato viejo y aburrido, no hay nada que aprender ahí."

"La transport layer es la capa externa: define por dónde viajan los mensajes — qué canal físico se usa y cómo se autentican. Y acá viene la propiedad clave: el mismo set de mensajes JSON-RPC funciona idéntico sobre cualquier transporte. Es lo que les da la portabilidad del protocolo. Ustedes no aprenden JSON-RPC para usar MCP — alcanza con saber que abajo hay un formato estándar."

Los dos transports definidos por la spec. Lo importante es la regla de cuándo usar cada uno. La mayoría de los servers populares hoy son stdio.

"Los dos transports que la spec define. Stdio — el server corre como proceso local en la misma máquina que el host. Los mensajes viajan por stdin y stdout. Es el modo el server vive con vos en tu máquina. Cero overhead de red. La mayoría de los servers populares hoy son stdio — filesystem, git, sqlite. Tu máquina los corre y listo."

"Streamable HTTP — el server corre remoto y el host se conecta por red. Usa HTTP POST para mensajes cliente a server, con Server-Sent Events opcionales para streaming. Soporta autenticación HTTP estándar: bearer tokens, API keys, OAuth. Sirve cuando el server vive en un servicio compartido — varios desarrolladores hablando con la misma instancia de Postgres centralizado — o cuando lo hostea un tercero, como GitHub o Notion que exponen su API por MCP."

"La regla simple es la del caption: disco o red local — stdio. Servicio compartido o de terceros — Streamable HTTP."

"La síntesis. Por qué tres primitives distintas y no una sola con flags. La razón es que cada una tiene un dueño distinto de la iniciativa, y esa diferencia se propaga al resto del diseño. "Iniciativa distinta lleva a permisos distintos lleva a interfaz distinta. Mezclarlas las tres en una sola primitive haría imposible un modelo de permisos coherente: el host no sabría cuándo pedir confirmación, cuándo cargar transparente, cuándo esperar al usuario. Por eso la spec separa."
