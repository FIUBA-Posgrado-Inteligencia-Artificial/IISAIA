Tres reveals — uno por capacidad faltante.

El alumno aterriza tres casos concretos que cualquiera vivió. Cada card tiene el pedido en mono (lo que vos querrías escribir) y por qué el set actual no lo cubre limpio.

"Tres ejemplos concretos de capacidad que el set por default no te da."

"Primero: querés tirar una query contra Postgres de producción y traerte filas que matchean un patrón. Bash con psql te lo permite si las credenciales están en el entorno. Pero el agente no sabe el schema, no sabe los tipos, no sabe qué columnas son sensibles. Querrías un tool con tipos declarados y permisos, no una shell ciega."

"Segundo: querés controlar un browser. Abrir una página, hacer click, sacar una screenshot, completar un form. Bash con curl no hace nada de eso — no renderiza, no ejecuta JS, no clickea. La capacidad no está."

"Tercero: querés que el agente use un SDK que salió ayer. El modelo tiene cutoff de entrenamiento. Si la API cambió, no se entera; sale de memoria y alucina firmas que no existen. Querrías que pueda consultar la doc real antes de tipear, no salir de memoria."

"Tres situaciones reales. Tres cosas que el agente, con el set actual, no puede. ¿Qué hacés?"

¿Cómo le damos Postgres al agente?

1

El runtime ya corre psql.

$ psql postgresql://readonly:s3cret@10.0.1.42:5432/analytics \
    -c "SELECT email FROM users LIMIT 5"

Funciona — mientras lo que Opus sabe sobre psql cubra lo que necesitás.

2

Agregás instrucciones al contexto.

# CLAUDE.md
## Base de datos
- Conexión: postgresql://readonly:s3cret@10.0.1.42:5432/analytics
- Schema: users(id, email, signup_date, active)
- Siempre limitá con LIMIT en queries exploratorias

Mejor. Pero ¿cómo sabés si no te falta algo? No sabés lo que no sabés.

3

Sumás rules y permisos para que no haga DELETE.

// .claude/settings.json
{
  "permissions": {
    "deny": ["Bash(psql*DELETE*)", "Bash(psql*UPDATE*)", "Bash(psql*DROP*)"]
  }
}

Robusto. Para Claude Code.

Cuatro reveals — el viaje real del dev.

Escenario concreto: vos sos un dev usando Claude Code con Opus, querés trabajar contra una base Postgres. Vas iterando con las herramientas que ya conocés de S04/S05 — Bash, CLAUDE.md, rules, settings — y descubrís que el problema estructural aparece cuando querés moverte a otro runtime.

"Ahora vamos al escenario real. Ustedes están usando Claude Code con Opus, y quieren que el agente trabaje contra una base de Postgres. ¿Cómo lo van resolviendo, paso a paso? Todo lo que vamos a ver son herramientas que ya conocen de las últimas dos clases."

"Paso uno. Lo más simple: el runtime ya tiene Bash, y Opus sabe algo de psql del entrenamiento. Le tiran un pedido, el agente arma una query psql y la ejecuta. Funciona — mientras lo que Opus sabe sobre psql cubra lo que ustedes necesitan. Si Opus se olvidó del flag para timestamps con timezone, o no conoce la convención que usa el equipo, ahí estás. Están a la merced de los pesos del modelo."

"Paso dos. Quieren más control. Agregan instrucciones al contexto — un CLAUDE.md con detalles de la base, el schema, las convenciones del equipo. Mejor. Pero acá aparece un problema sutil: cómo saben que no les falta nada. La doc la escriben ustedes, y ustedes no saben lo que no saben. Cada vacío en esa doc es un riesgo de que el agente actúe mal porque le falta información."

"Paso tres. Suman reglas y permisos. Configurás el settings.json de Claude Code para que el agente nunca corra DELETE, UPDATE o DROP sin pedir confirmación. Agregás hooks que validen comandos antes de ejecutarse. Ahora tu setup es robusto. Para Claude Code."

"Paso cuatro. Y acá está la trampa estructural. Mañana querés probar Cursor — mismo agente, otro runtime. Tu CLAUDE.md no se llama CLAUDE.md ahí, se llama .cursorrules. Tu settings.json no tiene equivalente directo, hay otra forma de configurar permisos. Tus hooks viven en otro lado. Codex CLI tiene su propio formato — AGENTS.md, config.toml. Opencode también — AGENTS.md, opencode.json. Cada runtime tiene su propia forma de implementar lo mismo. Y por cada capacidad nueva — browser, Slack, lo que sea — multiplicás el trabajo otra vez. La pregunta que sigue: ¿por qué crece tan feo?"

Un reveal — N×M visible primero, después aparece N+M al lado.

Convergencia visual del §1: cada celda de la matriz es lo que el dev acaba de armar para Postgres en Claude Code — el CLAUDE.md + las rules + los permisos. Cuatro capacidades por cuatro runtimes = dieciséis setups custom. Un protocolo común lo colapsa a N + M.

"Acá está la convergencia. Miren el grid de la izquierda. Cada celda es lo que ustedes acaban de armar para Postgres en Claude Code — el CLAUDE.md con instrucciones, las rules con permisos, los hooks. Cuatro capacidades por cuatro runtimes. Postgres en Claude Code, un setup. Postgres en Cursor, otro setup. Postgres en Codex CLI, otro. Postgres en Opencode, otro. Y así con browser, con docs, con Slack. Cuatro por cuatro: dieciséis setups completos, cada uno con sus formatos. Si mañana aparece un runtime nuevo, cuatro setups más. Si aparece una capacidad nueva, otros cuatro. Esto crece feo. Esto es N por M."

"Si en cambio metés un protocolo común al medio — esto es lo que aparece a la derecha — la matriz se colapsa. Cada capacidad habla el protocolo una sola vez. Cada runtime entiende el protocolo una sola vez. Cuatro más cuatro: ocho conexiones. Runtime nuevo: una integración, hereda todas las capacidades. Capacidad nueva: una integración, queda disponible en todos los runtimes. La matriz desaparece. Esto es N más M. Y ese protocolo común tiene nombre — viene ahora."

La salida estructural

MCP

Un protocolo común que el host habla, los proveedores hablan, y N×M se vuelve N+M.

Cómo está armado por dentro — viene ahora.

Bridge al resto del bloque. El nombre cae, sin desarrollar todavía. La frase central es la promesa estructural; las próximas secciones la desarman.

"Lo que falta no es otro wrapper. Es un protocolo común. Si tanto los hosts como los proveedores de capacidades acuerdan hablar el mismo formato, entonces vos escribís el server una vez y funciona en todos los hosts que hablen el protocolo. El problema N×M colapsa a N+M."

"Ese protocolo existe, se llama Model Context Protocol — MCP — y lo propuso Anthropic en 2024. Pero acá viene lo importante: no es un producto de Anthropic. Es un estándar abierto que adoptaron Cursor, Codex CLI, Opencode, y otros. Por eso lo vamos a estudiar — porque lo que aprendan acá viaja con ustedes a cualquier runtime que se sumen a usar."

"En la próxima sección vemos por qué tiene que ser protocolo y no librería, y por qué precedentes como HTTP y LSP cuentan acá."
