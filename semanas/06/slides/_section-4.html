"La doc de MCP define tres primitives del lado del server — tres tipos de cosa que un server pone disponible. Y la clave para entenderlas es el framing oficial: cada una tiene un dueño distinto de la iniciativa, alguien que decide cuándo se usa. Tools las invoca el modelo, resources las decide cargar la aplicación, prompts las pide el usuario. Esa diferencia organiza todo. Vamos una por una, y al cierre vemos las cuatro que MCP también define del lado del cliente — siete en total."

Tools.

primitive 1 — model-controlled

Funciones que el LLM puede invocar. El cliente las descubre primero, después el modelo decide cuál llamar. Ejemplo: Playwright MCP.

1
tools/list
↳
catálogo que devuelve el server — cada tool con nombre, descripción e inputSchema
browser_navigate inputs: url: string (required)
"Navigate browser to a URL"
browser_click inputs: element: string, ref: string
"Click on an element on the page"
browser_take_screenshot inputs: filename: string
"Take a screenshot of the current page"
2
tools/call(name: "browser_navigate", arguments: { url: "https://docs.anthropic.com" })
↳
el server ejecuta y devuelve
{ content: [ { type: "text", text: "Navigated to docs.anthropic.com — page snapshot..." } ] }
así se ve en Claude Code
> Abrí docs.anthropic.com y dame los headers principales
→ tool call: browser_navigate({ url: "https://docs.anthropic.com" })
⚠ Allow browser_navigate? [y / n]
✓ approved · result returned to model
claude > Los headers principales son: Introduction, Quickstart, Models, Guides, API Reference...

Tres reveals — discover, invoke, CC interaction.

Tools como flujo de 2 pasos con schema completo + cierre concreto que muestra cómo se ve en Claude Code. La confirmación antes de ejecutar visualiza el "acción con efecto" sin necesidad de etiquetarlo.

"Primera primitive: tools. Funciones que el LLM puede invocar. Para el ejemplo uso Playwright MCP — el server real de Microsoft que ya van a ver en el demo del §7."

"Paso uno. El cliente le pregunta al server qué tools tiene, con un tools/list. El server le devuelve un catálogo — esos tres ejemplos son tres tools reales de Playwright MCP: navegar a una URL, hacer click en un elemento, sacar screenshot. Cada entry trae el nombre, una descripción que el modelo va a leer, y un schema de parámetros. Ese catálogo es lo que el modelo termina viendo en su contexto."

"Paso dos. Durante la conversación, cuando el modelo decide que tiene que abrir la doc de Anthropic, mira el catálogo, identifica browser_navigate, arma los argumentos concretos con la URL, y emite un tools/call. El server real abre el browser headless, navega a esa URL y devuelve el snapshot de la página al modelo."

"Y así se ve esto desde Claude Code. El usuario pide algo en lenguaje natural — 'abrí docs.anthropic.com y dame los headers'. Claude identifica que tiene que usar browser_navigate y arma el call. Pero antes de ejecutar, Claude Code pide confirmación al usuario — porque los tools cambian el mundo, son acción con efecto. Una vez aprobado, el server ejecuta y la respuesta vuelve. El modelo después la usa para contestar."

Resources.

primitive 2 — application-controlled

Datos pasivos accesibles por URI. La aplicación los descubre y decide cuáles cargar al contexto. Ejemplo: Postgres MCP — expone el schema de cada tabla.

1
resources/list
↳
catálogo — cada resource con uri, name, mimeType
postgres://db-prod/users/schema mimeType: application/json
name: "users.schema" · description: "Schema for users table"
postgres://db-prod/orders/schema mimeType: application/json
name: "orders.schema" · description: "Schema for orders table"
postgres://db-prod/payments/schema mimeType: application/json
name: "payments.schema" · description: "Schema for payments table"
2
resources/read(uri: "postgres://db-prod/users/schema")
↳
el server consulta la base y devuelve
{ contents: [ { uri: "postgres://db-prod/users/schema", mimeType: "application/json", text: '{ "columns": [{ "name": "id", "type": "uuid" }, { "name": "email", "type": "text" }, ...] }' } ] }
así se ve en Claude Code
[ el usuario marca los schemas de users y orders como contexto (acción en la UI de Claude Code, no escribiéndolo al modelo) ]
→ Claude Code emite: resources/read("postgres://db-prod/users/schema") y orders/schema
✓ schemas inyectados al contexto, antes de que el usuario pregunte nada
> Escribime una query que traiga usuarios con pedidos sin pagar
claude > SELECT u.id, u.email FROM users u JOIN orders o ON o.user_id = u.id WHERE o.status = 'unpaid' (con los nombres de columnas reales, sin alucinarlos)

Tres reveals — discover, invoke, CC interaction.

Resources con Postgres MCP. El schema es un caso net-new sobre CC — CC ya lee archivos nativamente, pero no se puede conectar a una DB para introspeccionar schemas. Cronología clave: el read sucede ANTES de que el usuario pregunte algo. El modelo nunca decide — la app dispara el read por acción del usuario en la UI.

"Segunda primitive: resources. Datos pasivos accesibles por URI. Ejemplo real: Postgres MCP server, oficial de Anthropic — expone el schema de cada tabla de la base como un resource. Y noten que esto es net-new sobre CC: CC ya tiene herramientas para leer archivos del filesystem, pero no se puede conectar a una base de datos para introspeccionar schemas. Postgres MCP llena ese hueco."

"Paso uno. El cliente le pide al server el catálogo con resources/list. El Postgres MCP introspecciona la base — corre las queries de catálogo de Postgres — y devuelve una URI por cada tabla. Tres ejemplos: users.schema, orders.schema, payments.schema. Cada URI viene con mimeType application slash json porque el contenido es JSON estructurado del schema."

"Paso dos. La app decide cargar el schema de users. Emite resources/read con la URI concreta. El server consulta la base y devuelve el JSON del schema: las columnas, tipos, nullability, índices. La app lo inyecta al contexto del modelo."

"Y así se ve en Claude Code, con la cronología correcta: el usuario marca los schemas de users y orders como contexto — esto puede ser desde un panel de resources del lado, una @-mention de los recursos disponibles, o cualquier mecanismo de UI que la app les exponga. Es una acción en la UI, no consulta al modelo. Claude Code emite los reads y carga los schemas al contexto antes de que el usuario pregunte nada. Recién después el usuario hace la pregunta: escribime una query que traiga usuarios con pedidos sin pagar. El modelo arma el SQL con los nombres reales de las columnas — sin alucinarlos, porque los tiene en contexto. Eso es application-controlled. Nota al margen: algunos servers exponen resource templates — URIs parametrizadas como github slash issues slash llave-owner, llave-repo, llave-number — que la app expande con valores concretos en tiempo real."

Prompts.

primitive 3 — user-controlled

Templates reutilizables. El usuario los descubre y los invoca explícitamente con argumentos. Ejemplo: Git MCP.

1
prompts/list
↳
catálogo — cada prompt con nombre, descripción y arguments
review-pr arguments: diff: string (required)
"Review a pull request diff with structured feedback"
explain-commit arguments: sha: string (required)
"Explain what a commit changes and why"
summarize-changes arguments: since: string (required)
"Summarize repo changes since a given date or commit"
2
prompts/get(name: "review-pr", arguments: { diff: "<diff content>" })
↳
el server expande el template y devuelve
{ messages: [ { role: "user", content: { type: "text", text: "Review the following PR diff... <diff>" } } ] }
así se ve en Claude Code
> /review-pr
⌘ slash command — Claude Code llama prompts/get("review-pr", { diff: ... })
→ template expandido, mensaje listo enviado al modelo
claude > Found 3 issues:
  1. Missing test coverage in auth.ts:42
  2. SQL injection risk in users.ts:88
  3. Inconsistent error handling pattern...

Tres reveals — discover, invoke, CC interaction.

Prompts como flujo de 2 pasos + cierre concreto que muestra la sintaxis de slash command que el usuario tipea. Eso visualiza el "user-controlled" — el usuario invoca explícitamente.

"Tercera primitive: prompts. Templates reutilizables. Para el ejemplo uso un Git MCP — un patrón real que aparece en varios servers de la comunidad para acciones recurrentes sobre el repositorio."

"Paso uno. El cliente le pide al server qué templates tiene disponibles con prompts/list. El server le devuelve un catálogo: review-pr para revisar un pull request, explain-commit para entender qué hace un commit concreto, summarize-changes para resumir los cambios desde una fecha. Cada template tiene un nombre, una descripción, y un schema de argumentos."

"Paso dos. El usuario decide pedirle al agente que revise un PR. El host emite prompts/get con el nombre y los args. El server combina internamente instrucciones para el modelo — qué buscar, qué priorizar, qué formato de respuesta — junto con el diff que el usuario pasó, y devuelve todo armado como un mensaje listo en la estructura messages que ven en la response. No ejecuta nada, sólo arma el material."

"Y así se ve en Claude Code. El usuario tipea barra review-pr — exactamente como cualquier slash command. Si tipean barra solita en Claude Code y tienen un server MCP conectado, los prompts del server aparecen ahí, mezclados con los comandos nativos. Claude Code llama prompts/get internamente, el template se expande, se envía al modelo, y el modelo arma el review. Por eso user-controlled — el usuario invoca explícitamente, no se disparan solos."

Por qué la separación.

Cada primitive tiene un dueño distinto de la iniciativa. Eso determina todo lo demás.

Primitive
Quién decide
Permisos
Interfaz típica
tools
El modelo
De acción, con confirmación
Dialogo de confirmación
resources
La aplicación
De lectura, transparentes
Carga automática / picker
prompts
El usuario
Invocación explícita
Slash commands

Iniciativa distinta → permisos distintos → UI distinta. Mezclarlas haría imposible un modelo coherente.

Síntesis: el por qué pedagógico de las tres primitives. La diferencia de iniciativa determina permisos y UI. Esa es la razón estructural por la que MCP las separa.

"La síntesis. Por qué tres primitives distintas y no una sola con flags. La razón es que cada una tiene un dueño distinto de la iniciativa, y esa diferencia se propaga al resto del diseño. Los tools los decide el modelo, y eso implica un modelo de permisos de acción con confirmación. Los resources los decide la aplicación, eso implica permisos de lectura, transparentes, sin pedir nada. Los prompts los decide el usuario, eso implica invocación explícita, slash commands."

"Iniciativa distinta lleva a permisos distintos lleva a interfaz distinta. Mezclarlas las tres en una sola primitive haría imposible un modelo de permisos coherente: el host no sabría cuándo pedir confirmación, cuándo cargar transparente, cuándo esperar al usuario. Por eso la spec separa."

Y del otro lado: cuatro primitives del cliente.

El cliente también expone primitives hacia el server. La comunicación no es de una sola dirección.

Sampling

El server le pide al cliente: "pedile al LLM esto". Habilita workflows agentic sin que el server necesite SDK de modelo propio.

Elicitation

El server le pide al cliente: "preguntale al usuario X dato". Para confirmaciones, preferencias, datos faltantes.

Roots

El cliente le declara al server: "estos directorios son tu scope". Coordinación, no enforcement de seguridad.

Logging

El server manda mensajes de log al cliente. Para debugging y observabilidad.

Las nombramos para que sepan que existen. El detalle vive en la doc oficial.

Cierre de §4: client primitives. Cuatro nombres rápidos, sin profundizar. El alumno tiene que poder reconocerlas cuando aparezcan después.

"Y para cerrar el bloque, una observación que suele pasarse por alto: del otro lado, el cliente también expone primitives hacia el server. La comunicación no es de una sola dirección, hay cuatro mecanismos definidos."

"Sampling: el server le pide al cliente que le pida al LLM algo. Útil cuando el server quiere lógica agentic pero no quiere incluir un SDK de modelo propio — usa el LLM que el cliente ya tiene. Elicitation: el server le pide al cliente que le pregunte algo al usuario — confirmaciones, preferencias, datos que faltan a mitad de una operación. Roots: el cliente le declara al server qué directorios son su scope de trabajo. Es coordinación, no enforcement de seguridad — un server bien comportado los respeta, pero la spec no obliga. Logging: el server manda mensajes de log al cliente, para debugging."

"Las nombramos para que las reconozcan cuando aparezcan en docs de servers que vayan a usar. Profundizarlas a cada una requiere su propio bloque, y la doc oficial lo hace bien. Cerrando el inventario: tres primitives del server, cuatro del cliente, siete en total. Lo que sigue es algo conceptualmente más jugoso: cómo es que el modelo, cada vez que arranca una conversación, sabe qué tools tiene disponibles cuando los servers fueron escritos después de su cutoff date."
