"Arranquemos por el host. El host es la aplicación de IA con la que ustedes interactúan — Claude Code, Cursor, Visual Studio Code, Claude Desktop. Es la pieza que tiene la UI, que corre el LLM, que ustedes tipean. Es el contenedor de todo lo que viene. Adentro del host viven los clientes — esos rectángulos rayados que aparecen ahora. Cada cliente MCP es un componente del host que mantiene una conexión con UN servidor MCP. Acá ven dos clientes porque el host se va a conectar con dos servers distintos."

"Y a la derecha aparecen los servidores. Son programas externos al host, que viven como proceso aparte. Pueden correr locales — el primer ejemplo, Postgres, vive en la misma máquina y se conecta por stdio — o pueden correr remotos, como el segundo ejemplo, GitHub, que es un servicio externo y se conecta por Streamable HTTP. Entre cada cliente y su servidor viaja JSON-RPC, bidireccional. Esa es la arquitectura. Ahora vamos a las definiciones canónicas para fijarlas."

Las definiciones canónicas.

Tal cual aparecen en la doc oficial.

MCP Host

La aplicación de IA que coordina y administra uno o más MCP Clients.

MCP Client

Componente adentro del host que mantiene una conexión con UN MCP Server.

MCP Server

Programa que provee contexto a MCP Clients. Vive como proceso aparte, local o remoto.

Definiciones canónicas. No paráfrasis — tal cual aparecen en modelcontextprotocol.io. La que vale la pena destacar es la del Client porque la regla "uno por server" sale de ahí.

"Las definiciones tal cual están en la doc oficial. El Host coordina y administra clientes. Los Clients — en plural — viven adentro del host y cada uno mantiene una conexión con un único Server. El Server provee contexto, corre aparte, puede ser local o remoto."

"Notá la palabra clave de la definición del Client: 'un MCP Server', singular. No 'los servers', no 'varios servers'. Un cliente, una conexión, un server. Esa regla es lo suficientemente importante como para dedicarle el próximo slide."

"La regla. Si el host está conectado a tres servers, corren tres clientes adentro. No es uno solo multiplexando — son tres, cada uno con su conexión dedicada. Esa es la forma de pensar la relación cliente-servidor en MCP."

"¿Por qué importa? Porque cuando hablemos de capabilities, primitives, discovery, en los próximos slides, todo se piensa por par cliente-servidor. La conexión es la unidad de trabajo. Si lo piensan como N clientes para N servers, las propiedades del protocolo cierran solas. Si lo piensan como un cliente multiplexando, se confunden."

Dos capas.

MCP se divide en data layer (qué se intercambia) y transport layer (por dónde).

capa interna

Data layer

Define qué se intercambia: estructura de mensajes, métodos, primitives, lifecycle, notifications. Wire format: JSON-RPC 2.0.

capa externa

Transport layer

Define por dónde viajan esos mensajes: canal físico y autenticación.

El mismo set de mensajes funciona idéntico sobre cualquier transporte. No tenés que aprender JSON-RPC para usar MCP — sólo saber que está abajo.

Separación canónica data layer vs transport layer. La razón pedagógica importante: el mensaje es el mismo independiente del canal. Eso es lo que hace que el protocolo viaje entre transports.

"MCP se divide internamente en dos capas, y vale separarlas mentalmente. La data layer es la capa interna: define qué se intercambia entre cliente y servidor. Mensajes, métodos, las primitives que vamos a ver en el próximo bloque, el lifecycle de la conexión, las notificaciones. Wire format JSON-RPC 2.0 — formato viejo y aburrido, no hay nada que aprender ahí."

"La transport layer es la capa externa: define por dónde viajan los mensajes — qué canal físico se usa y cómo se autentican. Y acá viene la propiedad clave: el mismo set de mensajes JSON-RPC funciona idéntico sobre cualquier transporte. Es lo que les da la portabilidad del protocolo. Ustedes no aprenden JSON-RPC para usar MCP — alcanza con saber que abajo hay un formato estándar."

Dos transports.

Cuándo se usa cada uno.

local

Stdio transport

El server corre como proceso local en la misma máquina. Mensajes por stdin/stdout. Cero overhead de red.

remoto

Streamable HTTP transport

El server corre remoto y el host se conecta por red. HTTP POST + Server-Sent Events opcionales. Auth HTTP estándar — bearer, API keys, OAuth.

Regla simple: disco/red local → Stdio. Servicio compartido o de terceros → Streamable HTTP.

Los dos transports definidos por la spec. Lo importante es la regla de cuándo usar cada uno. La mayoría de los servers populares hoy son stdio.

"Los dos transports que la spec define. Stdio — el server corre como proceso local en la misma máquina que el host. Los mensajes viajan por stdin y stdout. Es el modo el server vive con vos en tu máquina. Cero overhead de red. La mayoría de los servers populares hoy son stdio — filesystem, git, sqlite. Tu máquina los corre y listo."

"Streamable HTTP — el server corre remoto y el host se conecta por red. Usa HTTP POST para mensajes cliente a server, con Server-Sent Events opcionales para streaming. Soporta autenticación HTTP estándar: bearer tokens, API keys, OAuth. Sirve cuando el server vive en un servicio compartido — varios desarrolladores hablando con la misma instancia de Postgres centralizado — o cuando lo hostea un tercero, como GitHub o Notion que exponen su API por MCP."

"La regla simple es la del caption: disco o red local — stdio. Servicio compartido o de terceros — Streamable HTTP."
