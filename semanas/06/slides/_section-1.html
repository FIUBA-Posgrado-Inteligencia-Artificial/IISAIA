"Tres ejemplos concretos de capacidad que el set por default no te da."

"Primero: querés tirar una query contra Postgres de producción y traerte filas que matchean un patrón. Bash con psql te lo permite si las credenciales están en el entorno. "Segundo: querés controlar un browser. Abrir una página, hacer click, sacar una screenshot, completar un form. Bash con curl no hace nada de eso — no renderiza, no ejecuta JS, no clickea. La capacidad no está."

"Tercero: querés que el agente use un SDK que salió ayer. El modelo tiene cutoff de entrenamiento. Si la API cambió, no se entera; sale de memoria y alucina firmas que no existen. Querrías que pueda consultar la doc real antes de tipear, no salir de memoria."

"Tres situaciones reales. Tres cosas que el agente, con el set actual, no puede. ¿Qué hacés?"

La primera salida vieja. El visual de 4 wrappers iguales en 4 hosts distintos muestra la duplicación. La frase N×M deja el término plantado para §2, que lo desarrolla.

"La primera respuesta instintiva — y la que más se ve en la práctica — es armarse el wrapper a mano. Un script de Bash que el agente ejecuta. Un binario chico que invoca tu API. Un agente paralelo que hace de puente. Funciona, sin duda. Lo usás en Claude Code y resuelve."

"El problema empieza cuando querés salir de Claude Code. El wrapper lo escribiste para este host — el formato de invocación, las convenciones de output, la forma en que el agente lo descubre, todo eso es específico de este runtime. Si mañana querés probar Cursor, el wrapper no se mueve con vos. Tenés que reescribirlo. Si querés probar Codex CLI, otra vez. Cada capacidad por cada host es trabajo nuevo. Eso es N×M. Si N son los hosts y M las capacidades, mantenerlo es insostenible."

La segunda salida vieja. Las tres complicaciones se nombran rápido — el alumno ya conoce context engineering y sabe por qué pegar de más no escala. La frase del takeaway ata explícitamente con S04: las herramientas de manejo de contexto existen para esto mismo.

"La segunda respuesta instintiva, todavía más vaga: pegás. Copiás el schema, copiás el HTML, copiás el README. Si lo necesitás una sola vez, está bien. Si lo necesitás todas las sesiones — y en un workflow real lo necesitás todas las sesiones — se cae rápido."

"Tres razones por las que se cae, y las tres las venís viendo desde la clase de harness y context engineering. Costos: cada token que pegás se factura, y se factura cada vez. Frescura: lo que pegás es una foto, y dos minutos después está desactualizada. Ruido: más texto degrada la atención del modelo — pegar de más empeora la respuesta, no la mejora."

"Las skills, los sub-agents, el harness — todo eso lo aprendieron justamente para no estar pegando manualmente. Lo que estamos viendo acá es que el patrón de 'pegar todo al prompt', incluso bien envuelto, tiene techo. Para muchas capacidades necesarias, no alcanza."

La salida estructural

MCP

Un protocolo común que el host habla, los proveedores hablan, y N×M se vuelve N+M.

Cómo está armado por dentro — viene ahora.

Bridge al resto del bloque. El nombre cae, sin desarrollar todavía. La frase central es la promesa estructural; las próximas secciones la desarman.

"Lo que falta no es otro wrapper. Es un protocolo común. Si tanto los hosts como los proveedores de capacidades acuerdan hablar el mismo formato, entonces vos escribís el server una vez y funciona en todos los hosts que hablen el protocolo. El problema N×M colapsa a N+M."

"Ese protocolo existe, se llama Model Context Protocol — MCP — y lo propuso Anthropic en 2024. Pero acá viene lo importante: no es un producto de Anthropic. Es un estándar abierto que adoptaron Cursor, Cline, Codex CLI, Visual Studio Code, Aider, Gemini CLI, y otros. Por eso lo vamos a estudiar — porque lo que aprendan acá viaja con ustedes a cualquier host que se sumen a usar."

"En la próxima sección vemos por qué tiene que ser protocolo y no librería, y por qué precedentes como HTTP y LSP cuentan acá."
