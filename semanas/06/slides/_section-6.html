Y el config se ve casi igual en todos.

Para meter un server, el host lee un JSON parecido a este.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp"]
    }
  }
}

Un nombre, un comando que lo arranca, y opcionalmente args y variables de entorno. Ese bloque, con variaciones cosméticas, es lo que ponés en Claude Code, en Cursor, en Cline.

El config canónico. La estructura es estándar: nombre, command, args. La idea: no hay que aprender un formato nuevo por host — es casi el mismo JSON en todos.

"Y para conectarlo, el config se ve casi igual en todos los hosts. Es un JSON con una clave mcpServers, y adentro, por cada server, un nombre — acá playwright —, el comando que lo arranca — npx —, y los argumentos. Eso es todo."

"Cada host tiene su propio archivo de settings y alguna variación cosmética, pero la estructura es esta misma. Si aprendiste a configurar un server en un host, ya sabés hacerlo en todos. No hay un formato nuevo que aprender por cada agente."

Qué te da, concretamente.

Tres consecuencias de que el server sea portable.

Amortización

Aprendés a usar un server una vez y te sirve en todos los hosts. Mismos tools, mismos nombres.

Portabilidad

Cambiar de host no te cuesta tu setup. Te llevás el JSON, lo pegás, y todo lo que tenías sigue andando.

Durabilidad

Lo que aprendiste no caduca cuando aparezca el próximo IDE con IA. Mientras hable MCP, la mecánica aplica.

Tres reveals — amortización, portabilidad, durabilidad.

Las tres consecuencias prácticas de la portabilidad, para vos como ingeniero. No son tres ideas distintas — son tres caras del mismo hecho: el server es independiente del host.

"Primera: amortización. El esfuerzo de aprender a usar un server se reparte entre todos los hosts. Si aprendiste a usar Playwright MCP en Claude Code, no lo reaprendés cuando pruebes Cursor. Mismo server, mismos tools, mismos nombres."

"Segunda: portabilidad. Si en algún momento querés probar otro agente, no perdés tu ecosistema. Te llevás el JSON de configuración, lo pegás en el host nuevo, y todo lo que ya tenías sigue funcionando. No empezás de cero."

"Tercera: durabilidad. Lo que aprendieron sobre MCP no se vuelve obsoleto cuando salga el próximo IDE con IA. Mientras los nuevos hablen el protocolo — y lo van a hablar, porque ahí está el ecosistema — todo esto sigue aplicando. Es conocimiento que no caduca."

Un reveal — primero el contraste, después la pregunta correcta.

El equívoco a desarmar: MCP no reemplaza las tools del host, las extiende. Las tools nativas (en CC: Read/Write/Edit/Grep/Glob/Bash) son parte del host, anteriores a MCP. Callback explícito al recap de §1. De acá sale la pregunta cero.

"Acá está el equívoco más común. La gente piensa que MCP reemplaza las tools del host. No. Cada host trae su propio set de tools nativas, anteriores a MCP, que son parte del host y no se agregan ni se sacan. En Claude Code son las que ya conocen del arranque de la clase: Read, Write, Edit, Grep, Glob, Bash. Eso no es MCP."

"Lo que MCP agrega es extensión: capacidades que el host no trae. Y cada extensión tiene un costo — un proceso aparte, otra dependencia, otra superficie de seguridad. Por eso la pregunta correcta antes de instalar cualquier MCP server es siempre la misma, y es la que más vale la pena que se lleven: ¿el host ya lo hace? Si sí, instalarlo duplica y agrega complejidad. Si no, extiende y vale la pena."

La pregunta cero aplicada concretamente a CC. La clasificación es RESPECTO a CC — no en abstracto. Izquierda: lo que CC ya hace nativo, instalar el MCP duplica. Derecha: lo que CC no hace, el MCP extiende. Los dos de la derecha que están en negrita conceptual — playwright y context7 — son los demos que vienen.

"Apliquemos la pregunta cero a servers que ya existen y son populares. A la izquierda, los que en Claude Code son redundantes, porque el host ya lo hace. Filesystem: Read y Write ya están. Git: Bash con git ya está, y de hecho es más flexible que el MCP. Fetch: Bash con curl ya está. Instalar cualquiera de estos en CC no te suma nada — duplica."

"A la derecha, los net-new: capacidades que CC no tiene de fábrica. Playwright, para manejar un browser. Context7, para traer docs de SDKs nuevos que están fuera del cutoff del modelo. Postgres, para conectarse a una base. Slack, para hablar con Slack. El github MCP, que agrega operaciones que el gh de línea de comandos no expone igual. Estos sí valen la pena. Y dos de ellos — playwright y context7 — son justo los que vamos a poner a andar en los demos."

¿Y por qué existen, entonces?

Porque otros hosts no tienen lo que CC trae de fábrica.

claude.ai (el chat) no tiene Bash, no tiene Read, no tiene Grep. Un filesystem MCP ahí es net-new: le da algo que el host no tenía.

Mismo binario, distinto valor según el contenedor.

Resuelve la paradoja aparente: si filesystem y git son redundantes, ¿por qué existen? Porque "redundante" es respecto a CC, no en abstracto. En un host más fino (el chat de claude.ai) son net-new. Esta es la frase que cierra el matiz de "build once integrate everywhere".

"Y acá puede saltar una duda razonable: si filesystem y git son redundantes, ¿por qué existen, por qué la gente los escribió? Porque redundante es respecto a Claude Code, no en abstracto. Hay hosts mucho más finos. El chat de claude.ai, por ejemplo, no tiene Bash, no tiene Read, no tiene Grep — no tiene nada de eso de fábrica. Un filesystem MCP en ese contexto es net-new: le da capacidades que el host no tenía."

"Y esta es la consecuencia fina del build once integrate everywhere. El server es el mismo en todos lados, pero su valor no. Un server vale lo que aporta sobre el host concreto que lo aloja. Mismo binario, distinto valor según el contenedor. Por eso la pregunta cero siempre es respecto al host donde estás parado."
